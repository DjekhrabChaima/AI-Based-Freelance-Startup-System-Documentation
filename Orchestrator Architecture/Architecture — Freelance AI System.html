Architecture Système

User → AI Agent → Orchestrateur → Tools

Kimi K2.5 · FastAPI · PostgreSQL · Tool Calling Loop

👤 USER
Utilisateur — Frontend
Interface web · POST /api/ai/chat

L'utilisateur écrit un message depuis l'interface. La requête est envoyée à l'API avec un session_id unique qui identifie sa conversation, permettant à l'orchestrateur de retrouver l'historique.

Envoie
{
 "session_id": "abc-123",
 "message": "Find React dev 5+ years",
 "user_id": 42
}
Reçoit
{
 "response": "J'ai trouvé 3 React devs...",
 "session_id": "abc-123",
 "tools_used": 2
}
message → agent IA
🤖 NODE B
AI Agent — Kimi K2.5 (LLM)
Hugging Face Novita · Hébergé externement · Pas sur ton serveur

Le cerveau du système. Il reçoit le contexte complet (historique + message + descriptions des tools), réfléchit et prend une seule décision à chaque tour : soit appeler un tool pour obtenir des données, soit répondre directement en texte. Il ne contrôle pas la boucle — c'est l'orchestrateur qui décide de le rappeler.

⚙️ NODE A
Orchestrateur — Chef d'orchestre Python
FastAPI · Coordonne l'agent et les tools · Ne pense pas lui-même
L'orchestrateur ne fait rien d'intelligent — il coordonne. Il reçoit la décision de l'agent IA, exécute le tool demandé, réinjecte le résultat dans l'historique, et rappelle l'agent. Il tourne en boucle jusqu'à recevoir une réponse texte finale.
🧠
memory { }
Stocke l'historique de chaque conversation par session_id. Dict Python en dev, Redis en prod.
📋
TOOL_SCHEMAS [ ]
Liste JSON des descriptions des tools. C'est ce que l'agent lit pour savoir quels outils existent.
🔀
execute_tool ( )
Mappe le nom du tool vers la vraie fonction Python. C'est le seul endroit qui parle à PostgreSQL.
🔁
while True loop
Rappelle l'agent à chaque tool_call. S'arrête quand l'agent répond en texte. Max 10 tours.
orchestrateur appelle les tools
🔧 TOOLS
Les 3 Tools — Fonctions Python → PostgreSQL
JSON Schema lu par l'agent · Fonction exécutée par l'orchestrateur
Un tool est composé de deux parties : un JSON Schema que l'agent lit pour décider quoi appeler, et une fonction Python que l'orchestrateur exécute. L'agent ne touche jamais la base de données — il demande, l'orchestrateur exécute.
TOOL 1
search_freelancers
Cherche des freelancers dans la DB selon des critères. Retourne une liste de profils correspondants.
L'agent l'appelle quand
"Trouve un React dev"
"Dev Django à Alger < 400€/j"
INPUT — reçu de l'agent
{
 "skills": ["React", "TypeScript"],
 "min_experience": 5,
 "location": "Alger",  // optionnel
 "max_rate": 400.0    // optionnel
}
OUTPUT — retourné à l'orchestrateur
[{
 "id": 42,
 "name": "Yassine B.",
 "rate": 300,
 "city": "Alger",
 "rating": 4.9
}, ...]
TOOL 2
get_profile
Récupère le profil complet d'un freelancer précis. Utilisé après search pour obtenir le détail d'un candidat.
L'agent l'appelle quand
"CV de Yassine ?"
"Il a des avis clients ?"
INPUT — reçu de l'agent
{
 "freelancer_id": 42,
 "include_reviews": true,
 "include_portfolio": true
}
OUTPUT — retourné à l'orchestrateur
{
 "name": "Yassine B.",
 "bio": "8 ans React...",
 "rating": 4.9,
 "reviews": [{...}],
 "portfolio": [{...}]
}
TOOL 3
calculate_score
Calcule un score de compatibilité entre un freelancer et les besoins du client. Pondère skills, expérience, rating et tarif.
L'agent l'appelle quand
"Classe les 3 meilleurs"
"Qui est le plus compatible ?"
INPUT — reçu de l'agent
{
 "freelancer_id": 42,
 "criteria": {
  "skills": ["React"],
  "budget": 350
 },
 "weights": [0.4,0.3,0.2,0.1]
}
OUTPUT — retourné à l'orchestrateur
{
 "freelancer_id": 42,
 "score": 87.4,
 "breakdown": {
  "skills":95, "exp":80,
  "rating":90, "rate":70
 }
}
résultat → boucle → agent
Boucle interne — while True · MAX 10 tours
L'orchestrateur tourne en boucle. À chaque tour il envoie le contexte à l'agent → l'agent décide → si tool_call : l'orchestrateur exécute et réinjecte le résultat → tour suivant. La boucle s'arrête quand l'agent retourne finish_reason : "stop".

Composants du microservice
⚡ FastAPI
Endpoint /api/ai/chat. Reçoit, valide avec Pydantic, déclenche l'orchestrateur.
⚙️ Orchestrateur
Chef d'orchestre. Tient memory{}, TOOL_SCHEMAS[], execute_tool(), boucle while True.
🤖 AI Client
Wrapper InferenceClient HF. Envoie le payload à Kimi K2.5 chez Novita.
🔧 Tool Functions
3 fonctions Python (search, get_profile, calculate_score). Chacune → SQL → PostgreSQL.
🧠 Memory
Dict Python memory{session_id:[messages]} en dev. Redis en production.
🗄️ PostgreSQL
Source de données des freelancers. Jamais appelée directement par l'agent.
